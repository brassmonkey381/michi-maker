# Architecture

A high-level map of how michi-maker fits together. For the domain model see
[DATA-MODEL.md](DATA-MODEL.md); for the database specifics see
[../supabase/README.md](../supabase/README.md).

## One codebase, three targets

michi-maker is a single [Expo](https://docs.expo.dev/) app that renders to **web, iOS, and
Android**. React Native primitives compile to native views on mobile and to DOM via
`react-native-web` in the browser. Where platforms must differ, we branch on `Platform.OS`
or use platform file variants (`*.web.tsx`, `*.ios.tsx`, `*.android.tsx`) — the nav rail
already does this (`src/components/nav/AppRail.tsx` vs `AppRail.web.tsx`).

```
              ┌────────────────────────────┐
   Web  ◄─────┤                            │
   iOS  ◄─────┤   Expo app (src/)          │
 Android ◄────┤   React + Expo Router      │
              └──────┬──────────────┬──────┘
        supabase-js  │              │  tcgscan-browse kit (HTTP, read-only)
             (typed) ▼              ▼
   ┌────────────────────────┐   ┌─────────────────────────────┐
   │ App backend (Supabase) │   │ tcgscan-data (Supabase)     │
   │ piikwvntldytjejxmcla   │   │ bmhjizcmwtmcrstadqto        │
   │ users, binders,        │   │ catalog, images, prices,    │
   │ entitlements + RLS,    │   │ similarity — public-read,   │
   │ edge fns (checkout,    │   │ populated by the            │
   │ webhook, sso, proxy)   │   │ tcgscan-data pipeline repo  │
   └────────────────────────┘   └─────────────────────────────┘
```

Two backends, deliberately separate: the **app backend** holds only user data (shared with
the sibling tcgscan-app — one account, one entitlements ledger), while **tcgscan-data** is
the read-only card catalog server owned by another repo (see [DATA-SERVER.md](DATA-SERVER.md)).
The web build deploys to **Vercel**, where `api/` adds serverless Open Graph preview
functions ([OPEN-GRAPH.md](OPEN-GRAPH.md)).

## Layers

### Routing & screens — `src/app/`

File-based routing via Expo Router. Each file is a route; `_layout.tsx` defines the navigator.
The `@/` path alias maps to `src/` (see `tsconfig.json`).

### UI — `src/components/`, `src/constants/`, `src/hooks/`

Reusable presentational components plus design tokens. `ThemedText` / `ThemedView` read from
`Colors` in `src/constants/theme.ts` for automatic light/dark support. Spacing and fonts are
tokenized there too.

### Data access — `src/lib/supabase.ts`

A single typed `supabase` client, configured once for both runtimes:

- **Native:** sessions persist in `AsyncStorage`; the client auto-refreshes the token while the
  app is foregrounded (driven by `AppState`).
- **Web:** sessions persist in `localStorage` and the auth redirect is read from the URL
  (`detectSessionInUrl`).

The client is parameterized by the `Database` type, so every query is type-checked end to end.

### Types — `src/types/`

- `database.ts` — the shape of the Postgres `public` schema (mirrors `supabase/migrations`;
  regenerate with `supabase gen types`).
- `domain.ts` — app-facing entity types derived from `database.ts`, plus presentation metadata
  such as the list of Michi layout styles. Components import from here, not from `database.ts`.

### Catalog access — `src/lib/catalog*.ts` + the `tcgscan-browse` package

The browse kit (CatalogBrowser, CardActionModal, query grammar, catalog/prices/similarity
clients) is the shared **`tcgscan-browse`** npm package. Thin app-side shims configure and
consume it: `catalogConfig.ts` (env → `configureBrowse()`), `catalog.ts` (`DemoCard`
adapter), `prices.ts` (binder/page value totals), `catalogSource.ts` (the gated encrypted
catalog download with a public fallback) and `catalogCache.ts` (encrypted-at-rest offline
cache). See [DATA-SERVER.md](DATA-SERVER.md) for the full seam table.

### Binder data layer — `src/data/`, `src/store/`

`src/data/binderTypes.ts` holds ergonomic view-models. Bundled example binders ship as
committed JSON (`generatedBinders.json`, `featuredBinders.json`, `releaseBinders.json`,
regenerated by the `scripts/build-*-binders.mjs` scripts) plus a small legacy sample set
(`sampleData.ts`, kept for the jumbo/V-UNION cards).

`src/store/binders.tsx` is a React context exposing binders + all CRUD / slot-editing actions over
React state (so the UI stays snappy). It is **backend-aware**:

- **Local mode** (no Supabase credentials): everything runs in-memory, seeded with the examples.
- **Cloud mode** (Supabase configured): on mount it ensures a session (signing in anonymously when
  there's none) and loads the user's binders; every mutation updates state immediately and persists
  in the background (optimistic). The bundled examples are never persisted — duplicating one creates
  a real user binder.

`src/data/binderRepo.ts` is the **only** module that talks to Supabase. It maps the flat
view-models ↔ the `binders` / `binder_pages` / `binder_slots` rows and runs the reads/writes
(scoped to the user by RLS). The screens/components in `src/components/binder/` consume the store
and are identical in both modes — which is why the view-models were shaped to mirror the schema.

### Backend — `supabase/`

Postgres schema, Row Level Security, and seed data as versioned SQL migrations. Auth is handled
by Supabase Auth; a trigger creates a `profiles` row on signup. **Every table has RLS enabled** —
the client can only ever read/write rows the signed-in user is entitled to. Beyond the binder
tables, the schema now carries entitlements/billing, the shared `user_cards` collection,
saved slices, print/PDF-snapshot bookkeeping, and PRO-trial/reclaim machinery — see
[../supabase/README.md](../supabase/README.md).

Server-side logic lives in `supabase/functions/` (Deno edge functions): `stripe-checkout`
(creates Checkout sessions), `payments-webhook` (Stripe fulfillment → entitlement grants),
`auth-handoff` (cross-app SSO, [AUTH.md](AUTH.md)), and `art-proxy` (allowlisted CORS image
relay for the fill-sheet PDF).

## Security model in one paragraph

The app ships only the **publishable** key, which is safe for clients because Row Level Security
decides what each user can see and do. Owners get full access to their own binders, pages, and
slots; anyone can read binders flagged `is_public`; the `entitlements` table has **no client
write policies** — grants come only from the service role (the payments webhook). The
`service_role` / secret key never appears in app code — it lives only in edge-function secrets.

## Data flow example — opening a binder

1. User signs in → Supabase Auth issues a JWT held by the client.
2. The binder screen queries `binders` → `binder_pages` → `binder_slots` through the typed
   client. Slots reference cards by plain-text id (no FK to a local `cards` table).
3. RLS filters rows to the owner (or to public binders) automatically — no `where owner_id = …`
   needed in app code, and none can be bypassed from the client.
4. Card images resolve straight from the card id against the tcgscan-data storage buckets
   (`cardThumbUrl(id, tier)` in `src/lib/catalogConfig.ts` — 245px webp for grids, 640px for
   binder pages, full-size jpg fallback).
